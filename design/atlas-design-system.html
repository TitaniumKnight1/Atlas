00 · principles

Design-system note

How to keep nine feature slices cohesive. Read this before building any module UI.

Principles

  • Elevation over borders. Separate surfaces by tone + soft shadow (base → panel → card), not heavy outlines. Borders are hairline and low-contrast. This is what makes dense screens feel integrated and smooth, not boxy.
  • One confident accent. A single saturated Atlas teal carries primary actions, active state, focus, and key data. Everything else is a calm grey text ramp so dense screens stay light.
  • Calm by default, bold in the signature. Boldness is concentrated in the pill + accent signature; all other surfaces are quiet.
  • Copy is design material. Name things by what the user does (Import project, Preview changes, Undo, Approve). Sentence case. Errors say what happened + how to fix. Empty states invite the next action.
  • Accessible & calm motion. Visible keyboard focus everywhere, AA contrast, transitions are short and respect prefers-reduced-motion.

The pill-vs-soft-rounded rule

The rounded-shape language is the cohesion device — but applied with restraint.

Pills · 999px buttons, status badges, tags, segmented controls, tabs, filter chips, toggles — controls & status
Soft · 12px text inputs, selects, textareas, cards, panels, table rows, large containers

Why: full-pill text fields read as search bars and hurt forms; rounded rows fight dense data. Pills are for controls + status; soft-rounded for inputs + containers.

Surface / accent logic

--bg-base app → --surface-1 panel → --surface-2 card → --surface-3 hover. Accent only for: primary action, active nav/tab, focus ring, selected row, key metric. Semantic colors (success / warn / danger / info) are reserved for state, never decoration.

Template — building a new feature slice

  1. Frame it in the shell. Add the module to the sidebar nav group; set breadcrumb + page title. Page = header row (title + primary pill action) then content.
  2. Use surfaces, not boxes. Group content in .panel/.card; rely on elevation. Never add heavy borders to separate regions.
  3. Controls & status are pills; inputs & data are soft. Buttons/segments/badges/chips = pill; forms/tables = md radius.
  4. Mutations go through the command panel. Any write reuses preview → dry-run → execute → undo. Dangerous/capability actions open the approval prompt or consent dialog.
  5. Cover the three states. Ship loading (skeleton), error (envelope code + fix), and empty/onboarding (invite next action) for every data view.
  6. Map feedback to severity. Use the alert/toast severities tied to the API error envelope (info/success/warn/danger).

14 · signature pattern

App shell — left sidebar + top bar

Decision: persistent left sidebar. A dense, multi-module tool with 10 feature areas needs a nav that scales vertically, supports grouping/labels and per-item status counts, and stays visible while deep in a module. Top-nav would crowd at 10 items; a thin icon rail would hide the module names this product leans on. The top bar carries project switcher context, breadcrumb, and the global command (⌘K).
