# Dependency Strategy

Phase 4 documents the intended toolchain and approval process. M0a introduced the first approved runtime manifests (`backend/requirements*.txt`, `package.json`, `src-tauri/Cargo.toml`).

## Intended Toolchain

From [docs/atlas-brief.md](../atlas-brief.md) and architecture docs:

### Frontend

- React
- TypeScript
- Tauri
- Vite
- TailwindCSS
- Monaco Editor
- xterm.js

### Backend

- Python 3.13
- FastAPI
- SQLAlchemy
- APScheduler
- GitPython
- Pydantic
- **Uvicorn** — ASGI server for the embedded FastAPI sidecar; required to run the local backend process inside Tauri.
- **PyInstaller** — packages the Python backend into a Tauri `externalBin` sidecar executable for desktop distribution without a user-installed Python runtime.

### Data And Testing

- SQLite (Atlas metadata)
- pytest (backend tests)
- Playwright (E2E tests)

### Packaging And CI

- GitHub Actions (described in [ci-cd-strategy.md](./ci-cd-strategy.md))

## Management Approach (M0a)

1. **Python**: `backend/requirements.txt` for runtime deps and `backend/requirements-dev.txt` for test/build tooling (`pytest`, `PyInstaller`).
2. **Node**: root and `frontend/package.json` with npm lockfiles.
3. **Rust (Tauri)**: `src-tauri/Cargo.toml` and `Cargo.lock` when generated by CI/local builds.
4. **Plugins**: plugin SDK dependencies must be declared in plugin manifests and reviewed separately from core dependencies.

## Approval Rules

- No new dependency without documenting why existing tools are insufficient.
- Prefer standard library or already-approved stack items before adding libraries.
- Security-sensitive areas (telemetry, filesystem, process control, plugins) require explicit review.
- Dependencies that imply network access or telemetry require privacy review.
- Avoid overlapping libraries for the same concern (e.g., multiple HTTP clients, multiple schedulers).

## Explicitly Deferred

- `pyproject.toml` with full dependency metadata (requirements files remain the M0a source of truth)
- Python package manager choice (`uv`, `poetry`, or plain `pip` + `venv`) beyond current pip usage
- Node package manager choice beyond current npm usage
- Tool configs not required for the M0a walking skeleton (ruff, eslint, mypy, Playwright config)

## Open Questions

- Python package manager: `uv`, `poetry`, or plain `pip` + `venv`?
- Node package manager: `npm`, `pnpm`, or `yarn`?
- Whether Monaco and xterm.js are bundled at frontend or loaded dynamically.
- Whether Sentry SDKs are added to frontend, backend, or both with separate sanitization paths.
